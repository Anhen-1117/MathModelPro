# 数学建模全流程设计思路

> 核心设计：5 角色协作流水线 + 统一模板系统 + 质量评估闭环

---

## 一、整体流程概览

```
用户输入                    AI 处理流水线                       输出交付
───────                    ────────────                       ────────

赛题 PDF/Word  ──→  ┌─────────────────────────────────┐  ──→  论文 Word
选择模板       ──→  │ 调度器 → 建模手 → 编程手 → 论文手 │  ──→  论文 PDF
注意事项       ──→  │         ↕          ↕          ↕    │  ──→  代码
                    │     知识库/文献   可视化/图表       │  ──→  可视化图表
                    │         ↓                         │
                    │      评估器 ←─────────────────────│  ──→  评分报告
                    └─────────────────────────────────┘
                                         │
                                   用户决断：保存/重做
```

## 二、五大角色定义

### 角色 1：调度器（Orchestrator）

**职责**：接收用户输入，拆解赛题，协调后续角色工作

| 能力 | 说明 |
|------|------|
| 文档解析 | 从 PDF/Word 提取赛题文本 |
| 问题拆解 | 识别问题类型（优化/预测/评价/分类），拆分子问题 |
| 需求分析 | 提取关键变量、约束条件、目标 |
| 模板选择 | 根据用户选择的赛事（CUMCM/MCM）加载对应模板 |
| 注意事项注入 | 将用户的特殊要求写入上下文，传递给后续角色 |
| 流程调度 | 按顺序调用建模手→编程手→论文手，管理检查点 |

**输入**：赛题文件 + 模板 ID + 用户注意事项
**输出**：结构化问题分析 + 子问题列表 + 建模建议

---

### 角色 2：建模手（Modeler）

**职责**：给出切题的数学模型

| 策略 | 说明 |
|------|------|
| LLM 生成 | 通过 DeepSeek 等大模型直接生成数学模型（主力方案） |
| 知识库检索 | 从内置 RAG 知识库匹配相似问题的模型（辅助） |
| 联网搜索文献 | 通过 OpenAlex / Tavily 搜索最新论文中的模型（补充） |

**核心原则**：

1. **切题第一**：无论模型从哪来，必须根据赛题实际情况修改适配
2. **文献溯源**：所有参考的文献必须列出，保存在文献目录中
3. **符号规范**：使用 LaTeX 格式，变量定义清晰
4. **假设合理**：明确模型假设和适用范围

**输入**：问题分析 + 子问题列表
**输出**：数学模型（LaTeX 公式）+ 符号说明 + 模型假设 + 参考文献列表

---

### 角色 3：编程手（Coder）

**职责**：算法实现，结果可复现，产出可视化

| 能力 | 说明 |
|------|------|
| 代码生成 | 通过 LLM 生成完整可执行的 Python 代码 |
| 算法优化 | 选择高效算法，添加中文注释 |
| 可复现性 | 固定随机种子，代码完整可独立运行 |
| 可视化 | 生成 matplotlib 图表，保存到指定目录 |

**核心原则**：

1. **结果可复现**：任何人拿到代码和数据，运行后得到相同结果
2. **算法切题**：不套用无关算法，每一步都与模型对应
3. **图表完整**：折线图、柱状图、热力图、流程图等按需生成
4. **注释清晰**：中文注释说明每一步的逻辑

**输入**：数学模型 + 数据
**输出**：可执行代码 + 运行结果 + 可视化图表文件

---

### 角色 4：论文手（Writer）

**职责**：在模板中填充论文，确保逻辑完整

| 能力 | 说明 |
|------|------|
| 模板填充 | 严格按照 CUMCM/MCM 模板的章节结构写作 |
| 公式引用 | 引用建模手给出的 LaTeX 公式 |
| 图表插入 | 引用编程手生成的可视化图表 |
| 逻辑连贯 | 确保摘要→问题重述→模型→求解→分析→结论全链路逻辑完整 |

**核心原则**：

1. **严格模板**：Word 和 PDF 使用同一套模板——修改模板，两端同步改变
2. **公式图表完整**：不遗漏任何一个模型公式和可视化图表
3. **上下文一致**：前后章节逻辑衔接，不出现矛盾
4. **参考文献规范**：格式统一，编号正确

**输入**：数学模型 + 代码 + 图表 + 模板
**输出**：完整 Markdown 论文

---

### 角色 5：评估器（Evaluator）

**职责**：对最终结果分析打分，给出参考意见

| 检查项 | 方法 |
|--------|------|
| 完整性 | 检查必要章节（摘要、模型假设、模型建立、结果分析、参考文献） |
| 公式质量 | 公式数量是否充足、格式是否正确 |
| 图表质量 | 是否有图表引用、图表是否清晰 |
| 代码可复现 | 检查代码是否可执行、输出是否与论文一致 |
| 逻辑一致性 | 模型→代码→论文三者是否一致 |
| 占位符检测 | 检查是否有"待补充"/"TODO"等未完成内容 |

**输出**：评分报告（0-100 分）+ 问题列表 + 改进建议

## 三、模板系统设计

```
                  paper_style.py（唯一配置源）
                        │
          ┌─────────────┼─────────────┐
          │             │             │
    页面设置        字体配置        段落格式
    A4 2.5cm      Times New Roman  首行缩进2em
                  宋体 12pt        两端对齐
                                  行高1.8
          │             │             │
          ├─────────────┼─────────────┤
          │             │             │
     三线表样式     代码块样式      标题样式
     顶底1.5pt     灰底#f2f2f2     h1=17pt居中
     表头0.75pt    边框0.8pt       h2=14pt
                                   h3=12pt
          │                         │
          ▼                         ▼
     docx_exporter.py        PaperPreview.vue
        (Word)               (PDF 浏览器打印)
```

**关键设计**：修改 `paper_style.py` 中任意一个值，Word 和 PDF 同时生效，不需要改任何其他文件。

## 四、数据流与状态管理

```
TaskDB 一条记录的生命周期：

pending ──→ running ──→ completed ──→ 用户决断
              │                         │
              │  ┌──────────────────────┤
              │  │ 各阶段进度实时更新     │  保留或
              │  │ progress: {          │  重新生成
              │  │   analysis: 100,     │
              │  │   modeling: 100,     │
              │  │   coding: 100,       │
              │  │   paper: 100,        │
              │  │   overall: 100       │
              │  │ }                    │
              │  └──────────────────────┤
              │                         │
              ▼                         ▼
          failed                   重新进入 running
     （LLM 调用失败等）
```

## 五、质量闭环

```
                    ┌──────────────┐
                    │   用户输入     │
                    │ 赛题+模板+备注 │
                    └──────┬───────┘
                           │
              ┌────────────▼────────────┐
              │  调度器：拆解问题         │
              └────────────┬────────────┘
                           │
              ┌────────────▼────────────┐
              │  建模手：生成模型         │←── 知识库
              │  （列出所有参考文献）     │←── 联网搜索
              └────────────┬────────────┘
                           │
              ┌────────────▼────────────┐
              │  编程手：算法+可视化      │
              │  （结果必须可复现）       │
              └────────────┬────────────┘
                           │
              ┌────────────▼────────────┐
              │  论文手：模板填充         │
              │  （Word+PDF 同模板）     │
              └────────────┬────────────┘
                           │
              ┌────────────▼────────────┐
              │  评估器：打分+建议        │
              │  完整性/公式/图表/逻辑    │
              └────────────┬────────────┘
                           │
                    ┌──────▼──────┐
                    │  用户决断    │
                    │  保存 or 重做 │
                    └─────────────┘
```

评估器的建议 + 用户自己的感觉 → 最终决定。用户是最终决策者。
